## 3. Correção inicial de migrations

O Docker inicialmente retornava HTTP 500 porque o código conhecia novos campos,
mas o banco local ainda estava em uma revisão anterior. `db.create_all()` cria
tabelas ausentes, porém não atualiza tabelas já existentes.

O ajuste introduziu:

- validação fail-closed da fundação já existente;
- migration aditiva dos campos operacionais e auditoria;
- `SKIP_SCHEMA_BOOTSTRAP=1` para separar Alembic do bootstrap legado;
- execução de `flask db upgrade` no entrypoint local.

As migrations históricas mantêm seus nomes originais:

- `9b7d4e2c1a60_integration_v2_foundation.py`;
- `c4a8f2d19e31_integration_v2_operational_review.py`.

Na primeira etapa, a passagem até `c4a8f2d19e31` preservou 161 lojas, 617
tarefas, 3.401 eventos de status e 202 bloqueios.

## 4. Consolidação do módulo de Integração

Uma auditoria mostrou que o legado ainda era consumido por Performance, Jarvis,
sincronização e banco. Por isso, a remoção foi feita somente depois da migração
desses consumidores.

O módulo ativo passou a usar:

- frontend em `frontend/src/features/integration/`;
- páginas `IntegrationMonitor` e `IntegrationAnalytics`;
- backend `/api/integration`;
- `IntegrationQueryService` e `IntegrationSyncService`;
- tabelas `integration_*`;

As rotas antigas não permanecem como aliases silenciosos: o smoke test confirmou
HTTP 404. O valor legado `INTEGRATION_V2` ainda é aceito apenas ao exibir logs
históricos; novos eventos usam `INTEGRATION`.

## 7. Segurança, RBAC e deploy

As leituras da Integração exigem autenticação. Operações sensíveis usam as
permissões centrais `sync_clickup`, `manage_performance`, `edit_store`,
`delete_store` e `manage_system`, conforme o domínio. Usuários inexistentes ou
inativos são rejeitados pela autenticação central.

O Render passou a usar `backend/production-entrypoint.sh`, que:

- interrompe o deploy se `MIGRATION_DATABASE_URL` estiver ausente;
- executa Alembic com conexão direta/session e bootstrap desativado;
- inicia Gunicorn somente após o upgrade bem-sucedido;
- mantém `DATABASE_URL` separada para o runtime.

A senha do banco e um webhook foram expostos durante a sessão e precisam ser
rotacionados antes de produção. Nenhum valor sensível é repetido nestes docs.

## 9. Produção/Supabase

Nenhum backup novo de produção foi concluído e nenhuma migration foi aplicada.
A credencial de banco compartilhada durante a sessão deve ser rotacionada.

Antes do deploy:

1. rotacionar credencial de banco e webhook;
2. configurar `MIGRATION_DATABASE_URL` direta/session na porta `5432`;
3. gerar backup novo de roles, schema e dados;
4. testar a restauração;
5. executar preflight somente leitura e registrar contagens;
6. validar em staging/cópia restaurada;
7. somente então acionar o deploy Render e os smoke tests.

Não usar o pooler transacional `6543` para migrations ou backup. O
[runbook](./MIGRATION_RUNBOOK.md) contém o procedimento completo.
